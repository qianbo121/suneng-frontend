# 官网（manufacturing-corporate-site）技术债务修复方案

> 用法：下面拆成 **5 个 Codex Prompt + 1 个你/运维亲自做的任务（部署）**。按顺序一个一个发，每个发完看到「验收通过 / 是否影响功能 YES-NO」再发下一个。不要一把梭。

## 技术栈
- backend：NestJS + Prisma + JWT/Passport + Swagger
- frontend：Next.js（app router、`[locale]` + next-intl）+ Tailwind
- admin：React + Vite + react-router + react-quill
- monorepo（pnpm workspace）；**0 测试，CI 只部署**

## 债务清单（按优先级，已逐条核实）
- **P0 安全/上线**：S1 CI 用错 compose（用 dev compose，JWT 默认 change-me，Postgres 暴露，停机）｜S2 RBAC 全局失效（RolesGuard 无 @Roles 即放行，全后端仅 1 处用 @Roles）｜S3 富文本 XSS（正则手搓 sanitizer 可绕过）｜S4 SVG/PDF 上传 XSS（仅校验客户端 MIME + 同源 inline）｜S5 Swagger 生产暴露｜S6 限流进程内 Map / trust proxy 未开 / JWT 存 localStorage
- **P1 功能**：B1+B2 i18n 死链（8 个 zh-only 页 + 正文 103 处硬编码 /zh/）｜B3 news `no-store` 抵消 ISR｜B4 sitemap 重复 + 每次抓 100 条｜B6 admin SWR error 被吞（空表冒充错误）｜B7 news viewCount 客户端写 + slug 竞态
- **P2 结构**：CMS 解耦（后端接口已齐全，前端 hook stub 死）｜admin ~60% 死代码｜210MB 图片进 git（.git 395MB）｜后端字段逐个手抄、跨模块复制

## 通用约束（每个 Prompt 都默认适用）
- 在**独立分支**做；开工前确认工作区干净（现有 backend/admin 未提交 WIP 先提交或 stash）。
- **不改 API contract、不删功能、不改页面视觉。**
- 每批结束必须：三包 `lint + tsc --noEmit + build` 全过 + 关键页**截图对比** + 输出「是否影响功能 YES/NO」。
- 提交规范：`<type>(scope): xxx`，一批一主题，别混。

## Prompt ②（批1A）安全：富文本 / 上传 / Swagger

```text
任务:官网安全收口(批1A)。分支 fix/security-batch1。不改业务/视觉/API。
修 3 项:
1) 富文本 XSS:frontend/src/lib/sanitize.ts 的正则 sanitizer 可绕过。引入 isomorphic-dompurify(或 dompurify),把 sanitizeRichTextHtml 改为基于 DOMPurify 的白名单清洗;所有用 dangerouslySetInnerHTML 渲染 admin 富文本的入口(NewsArticleContent 等)统一走它;后端 news 写入时也清一遍(纵深)。不改可见排版。
2) SVG/PDF 上传 XSS:后端 upload 仅校验客户端 mimetype(可伪造)。改为:server 端按 magic bytes 校验真实类型;svg/pdf 不再 inline——main.ts 的 useStaticAssets 给上传目录响应加 Content-Disposition: attachment(或对 svg/pdf 单独路由强制下载)+ 基本 CSP;图片保持 inline。
3) Swagger 生产关闭:main.ts 的 SwaggerModule.setup('api/docs',...) 用 NODE_ENV 守卫,仅 development 启用。
验收:三包 tsc+lint+build 过;手测新闻详情排版不变、图片正常、上传 svg 触发下载、生产构建 /api/docs 404。输出 before/after 安全对比 + 是否影响功能 YES/NO。
提交分三次:fix(web): replace regex sanitizer with DOMPurify / fix(api): validate upload by magic bytes & force-download svg-pdf / fix(api): disable swagger in production。
```

## Prompt ③（批1B）RBAC 收口 —— 高危，单独发，务必小步

```text
任务:官网后端 RBAC 收口(高危,单独一批)。分支 fix/rbac。
已核实:RolesGuard 是全局 APP_GUARD;当前 `if(!roles) return true` 等于不设防;且它不认 @Public(只看 @Roles)。公开接口(news/about/banner/contact/login 等)靠 @Public 跳过 JwtAuthGuard,但 RolesGuard 不看 @Public。
红线:绝对不能简单改成 `if(!roles) return false` —— 那会让所有 @Public 接口在 RolesGuard 这关 403,整站前台 + 后台登录全挂。
按序做:
1) 先让 RolesGuard 认 @Public:用 reflector 读 IS_PUBLIC(handler+class),是 @Public 直接 return true。
2) 再扫描所有非 @Public 的 admin 接口,逐个加 @Roles([...]):产品/新闻/分类/banner/about/strength/seo/delivery/certificate/上传/删除联系人留言 等内容写操作 → super_admin+editor;仅 super_admin 的(admin-user 账号管理)→ super_admin。
3) 然后把 RolesGuard 默认行为收紧为:@Public→放行;有 user 但无 @Roles 的非公开接口→拒绝。
验证(无测试,手动+起服务):公开站所有页面正常打开;super_admin 后台全部增删改正常;若有 editor 账号验证它能写内容、进不了 admin-user(403);contact/news 公开接口仍 200。
输出:加了 @Roles 的接口清单、RolesGuard before/after、公开接口验证结果、是否影响功能 YES/NO。
提交:fix(api): enforce RBAC default-deny while honoring @Public。
```

## 附：S1 部署修正 —— 不要让 Codex 盲改，你/运维亲自做

**为什么不发给 Codex**：改部署管道要知道服务器上 prod env 文件在哪、线上到底走 CI 还是手动 `deploy.sh`；Codex 没有这些事实，盲改可能让线上发不出去或停机。而且"修 JWT_SECRET=change-me 启动失败"的**正解是 prod 配真 secret，不是动那个守卫**（守卫是对的，别删）。

**你/运维按这个 checklist 做**：
1. 确认线上实际部署方式：是 GitHub Actions `deploy.yml`，还是手动 `deploy.sh`？
2. 把 `deploy.yml` 的 `docker compose down && up -d --build` 改为：用 `-f docker-compose.prod.yml --env-file <prod env>`，去掉 `down`（改滚动 `up -d`），与 `deploy.sh` 统一成一条 prod pipeline。
3. 在服务器 prod env 里配置真实 `JWT_SECRET` / `DB_PASSWORD`（**保留** `configuration.ts` 对 change-me 的启动拒绝守卫）。
4. 确认 prod compose 不把 Postgres `5432` 暴露宿主、用 `nginx.prod.conf.template`（带 TLS/域名）。
5. 改完先在 staging / 一次低峰部署验证，再常态化。
